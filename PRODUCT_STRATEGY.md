# CUA-Lark 产品策略文档

> 从产品思维出发，回答四个核心问题：谁会用？为什么用？怎么满足？如何落地？

---

## 一、潜在用户

### 1.0 先说清楚：谁不是我们的用户

> **CUA 不快。** 每轮迭代 3-5 秒（截图→AI思考→执行），一个简单任务可能耗时 30 秒到 2 分钟。
> 如果你对飞书操作滚瓜烂熟、手指比脑子快，那你自己点一定比 CUA 快。
> **CUA 不是给效率达人用的，是给"不会点"和"懒得点"的人用的。**

```
              CUA 比人快？                     CUA 比人慢，但...
                   │                                  │
     ┌─────────────┴─────────────┐      ┌────────────┴────────────┐
     │                           │      │                        │
  ✘ 飞书熟练用户              ✘ 效率狂    ✔ 不想学飞书的人     ✔ 不想动手的人
    "我 3 秒约完会"         "别浪费我时间"  "按钮在哪我都找不到"  "能不动手就不动手"
```

### 1.1 用户画像总览

```
                    ┌──────────────────────────────────────┐
                    │         CUA-Lark 真正的目标用户        │
                    │                                      │
                    │   ┌──────────┐    ┌──────────┐       │
                    │   │ 飞书小白  │    │ 极懒用户  │       │
                    │   │ (不会点)  │    │ (不想点)  │       │
                    │   └────┬─────┘    └────┬─────┘       │
                    │        │               │              │
                    │   ┌────┴───────────────┴─────┐       │
                    │   │      人机分离场景          │       │
                    │   │  (人在外面 / 电脑不在身边)  │       │
                    │   └──────────────────────────┘       │
                    └──────────────────────────────────────┘
```

### 1.2 三类核心用户

#### 🎯 用户 A：飞书小白（不会点）
| 维度 | 描述 |
|------|------|
| **身份** | 传统行业转型者、年长员工、新入职员工、对软件有畏难情绪的人 |
| **日常** | 被迫使用飞书办公，每次操作都要犹豫半天 |
| **痛点** | 找不到按钮在哪、不知道功能叫什么名字、怕点错不敢乱试 |
| **典型场景** | "领导让我在云文档里建个表格……云文档在哪？怎么建？" |
| **核心诉求** | 像跟同事求助一样跟 AI 说话，AI 替我把事办了 |
| **为什么不用手点** | 不是不想，是真不会。找按钮的时间比 CUA 慢多了 |

#### 🎯 用户 B：极懒用户（不想点）
| 维度 | 描述 |
|------|------|
| **身份** | 能躺着绝不坐着的效率"反向"追求者、能少点一下绝不多点一下的人 |
| **日常** | 会用飞书，但觉得操作流程太啰嗦，能省则省 |
| **痛点** | "预约会议要点 8 下，烦死了"；"群公告那个入口藏那么深，每次找半天" |
| **典型场景** | 躺在沙发上用手机 @ 机器人："把昨天那份纪要发到项目群" |
| **核心诉求** | 动嘴不动手。一句话说完，剩下的 AI 搞定 |
| **为什么不用手点** | 会点，但不想点。CUA 慢就慢，反正又不是我在等 |

#### 🎯 用户 C：人机分离者（点不了）
| 维度 | 描述 |
|------|------|
| **身份** | 远程办公者、出差人员、通勤路上、在家但电脑在另一个房间的人 |
| **日常** | 人不在电脑前，但需要操作电脑上的飞书 |
| **痛点** | 手机端飞书功能受限，很多操作只能在桌面端完成 |
| **典型场景** | 人在高铁上，手机飞书 @ 机器人："帮我把桌面上的报价单发给王总" |
| **核心诉求** | 手机就是遥控器，电脑在办公室自动干活 |
| **为什么不用手点** | 想点也点不了，人根本不在电脑旁边 |

### 1.3 用户价值公式

```
传统工具价值 = 完成任务的速度
CUA 的价值   = 完成任务 - 用户需要付出的操作

对飞书小白：  自己操作 = 不会 → 价值 = ∞（不用 CUA 根本完不成）
对极懒用户：  自己操作 = 烦 → 价值 = 省心（慢一点无所谓）
对人机分离：  自己操作 = 不可能 → 价值 = ∞（不用 CUA 根本做不到）
对熟练用户：  自己操作 = 快 → 价值 = 负（CUA 反而更慢）← 不是目标用户
```

## 二、用户需求

### 2.1 需求层次模型

```
        ┌──────────────────────────────────┐
        │  第 3 层：人机分离                  │
        │  "人在外面，电脑在办公室，照样操作"   │
        │  ──────────────────────────────── │
        │  第 2 层：不想动手                  │
        │  "我知道怎么点，但我就是不想点"       │
        │  ──────────────────────────────── │
        │  第 1 层：不会操作                  │
        │  "飞书那些按钮到底在哪？帮我点"       │
        └──────────────────────────────────┘
```

### 2.2 四大核心需求

#### 需求 1：零学习成本
> "我不想知道飞书怎么用，我就想让它帮我把事办了。"

- **现状**：飞书功能 100+，入口分散在多层菜单，新手根本找不到
- **典型场景**：领导说"在云文档建个表格"，用户连云文档在哪都不知道
- **期望**：用日常语言说需求，AI 自己找到功能入口并完成操作
- **对应人群**：飞书小白

#### 需求 2：动嘴不动手
> "预约会议要点 8 下，我不想点，你帮我点。"

- **现状**：很多飞书操作流程冗长（预约会议 6-8 步、发群公告 5-6 步）
- **典型场景**：躺在沙发上不想动，或者正在开会腾不出手
- **期望**：一句话说完，AI 在后台慢慢点，用户该干嘛干嘛
- **对应人群**：极懒用户
- **关键认知**：用户不在乎 CUA 慢，因为"等待"的成本由 AI 承担，用户不需要盯着看

#### 需求 3：远程遥控
> "人在高铁上，需要操作办公室电脑上的飞书。"

- **现状**：手机端飞书功能受限，很多操作只能在桌面端完成
- **典型场景**：出差路上需要发文件、通勤时需要查文档、在家需要操作公司电脑
- **期望**：手机就是遥控器，发条消息，电脑自动执行
- **对应人群**：人机分离者

#### 需求 4：过程不操心
> "AI 执行的时候不用我盯着，出问题了它会问我。"

- **现状**：传统自动化工具是"全自动黑盒"，出错了用户不知道
- **典型场景**：AI 执行中遇到不确定的情况（缺时间、缺人名）
- **期望**：AI 主动通过飞书消息问用户，用户回复后继续；不需要打开任何额外窗口
- **对应人群**：所有三类用户

### 2.3 需求 vs 用户矩阵

| 需求 \ 用户 | 飞书小白 | 极懒用户 | 人机分离者 |
|-------------|:------:|:------:|:--------:|
| 零学习成本 | ★★★★★ | ★★★ | ★★ |
| 动嘴不动手 | ★★★ | ★★★★★ | ★★★ |
| 远程遥控 | ★★ | ★★★ | ★★★★★ |
| 过程不操心 | ★★★★★ | ★★★★ | ★★★★ |

### 2.4 关键洞察：慢不是问题

```
用户感知的不是"AI 执行有多快"，而是"我需要付出多少操作"。

┌─────────────────────────────────────────────────────────┐
│                                                         │
│   自己操作飞书：  找入口(10s) → 犹豫(5s) → 点击(1s)      │
│                  → 填表(20s) → 检查(5s) = 41 秒 + 焦虑   │
│                                                         │
│   CUA 替你操作：  说一句话(5s) → 等结果(60s)             │
│                  = 65 秒 + 零焦虑 + 可以做别的事          │
│                                                         │
│   对小白来说：41 秒是"找不到"的 41 秒，实际可能 5 分钟     │
│   对懒人来说：41 秒是"不想干"的 41 秒，每一秒都是折磨      │
│   对人机分离：根本没法自己操作，CUA 是唯一解               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 三、如何满足

### 3.1 产品核心主张

> **"你不会点，或者不想点——那就别点。跟我说，我来。"**

三个关键词：
- **跟我说** — 自然语言，在飞书里 @ 一下就行
- **别点** — 承认 CUA 比熟练用户慢，但目标用户本来就不想/不会点
- **我来** — AI 真的替你操作桌面，不只是聊天

**坦诚面对速度**：
- CUA 不快，每轮 3-5 秒，一个任务 30 秒到 2 分钟
- 但目标用户自己操作更慢（小白找不到按钮）或根本不想动（懒人）或动不了（人机分离）
- **CUA 卖的不是速度，是"你不用动手"**

### 3.2 五大产品策略

#### 策略 1：AI 视觉驱动 —— "让 AI 看见屏幕"

```
传统自动化：预设坐标 → 机械点击 → 换个分辨率就崩
CUA-Lark：  截图 → AI 看图理解 → 自主决策坐标 → 执行
```

| 对比维度 | 传统 RPA | CUA-Lark |
|----------|----------|----------|
| 定位方式 | 硬编码坐标 | AI 视觉识别 |
| 分辨率变化 | 崩溃 | 自适应 |
| UI 改版 | 需重写脚本 | AI 自己找 |
| 新场景 | 需编程 | 自然语言即可 |

**核心价值**：用户不需要"编程"，只需要"描述"。AI 通过截图看见屏幕，自己判断按钮在哪、输入框在哪、下一步该做什么。

#### 策略 2：飞书原生入口 —— "不离开飞书"

```
传统工具：打开新软件 → 登录 → 学习界面 → 输入指令
CUA-Lark：在飞书里 @ 机器人 → 说需求 → 等结果
```

**核心价值**：用户已经在飞书里了，不需要切换到另一个工具。飞书就是操作界面。

#### 策略 3：问答式执行 —— "AI 不懂就问"

```
传统自动化：按预设流程走 → 遇到异常 → 报错退出
CUA-Lark：  执行中遇到不确定 → 飞书消息问用户 → 用户回复 → 继续
```

**核心价值**：AI 不是"全自动黑盒"，遇到不确定的事情会主动问。用户始终在决策链中。

#### 策略 4：迷你控制条 —— "看得见，不碍事"

```
传统方案：主窗口一直显示 → 遮挡屏幕 → AI 截图时看到自己的窗口 → 混乱
CUA-Lark：主窗口隐藏 → 左下角迷你条 → 显示状态 + 暂停/继续按钮
```

**核心价值**：用户知道 AI 在干什么（状态可见），但 AI 的"眼睛"（截图）不会被自己的 UI 干扰。

#### 策略 5：上下文连续记忆 —— "AI 记得之前做了什么"

```
传统方案：每次任务独立 → 上一个任务的经验全部丢失
CUA-Lark：对话历史保留 → 跨任务记忆 → 用户补充的信息 AI 一直记得
```

**核心价值**：不是"一次性工具"，而是"越来越懂你的助手"。

### 3.3 用户体验闭环

```
    ┌─────────────────────────────────────────────────────┐
    │                                                     │
    │   用户说需求                                         │
    │      │                                              │
    │      ▼                                              │
    │   AI 理解确认 ──→ 不确定？──→ 飞书消息问用户          │
    │      │                    ↑          │              │
    │      │ 确认               │          │ 用户回复      │
    │      ▼                    │          ▼              │
    │   开始执行                └──── 用户确认/补充         │
    │      │                                              │
    │      ▼                                              │
    │   截图 → AI 决策 → 执行操作 → 检查结果               │
    │      │         ↑                    │               │
    │      │         └── 用户可随时暂停 ──┘               │
    │      ▼                                              │
    │   任务完成 → 飞书通知用户                            │
    │                                                     │
    └─────────────────────────────────────────────────────┘
```

### 4.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| AI 模型 | 豆包 2.0 Pro（火山方舟） | 支持视觉理解，JSON 输出稳定，中文优化 |
| 桌面操作 | PyAutoGUI + pyperclip | 跨平台鼠标键盘控制，剪贴板操作成熟 |
| GUI 框架 | Tkinter（Python 内置） | 零额外依赖，Windows 原生渲染 |
| 飞书集成 | 飞书开放平台 WebSocket | 官方 SDK，长连接实时推送 |
| 截图 | PIL / MSS | 全屏截图，不依赖特定窗口 |
| 文档生成 | docx (npm) | 纯 JavaScript 生成 .docx，无需 Office |

```
        └────────────────────────┴───────────────────────┘
              所有跨线程 GUI 操作都通过 after() 投递
```

### 4.4 部署方案

#### 方案 A：单机部署（当前）

```
┌─────────────────────────────────────────┐
│          用户电脑 (Windows)              │
│  ┌─────────────────────────────────┐    │
│  │  python feishu_ws_client.py     │    │
│  │  ├─ GUI (tkinter)               │    │
│  │  ├─ WebSocket 长连接             │    │
│  │  └─ CUA 任务执行                 │    │
│  └─────────────────────────────────┘    │
│              │                           │
│              ▼                           │
│  ┌─────────────────────────────────┐    │
│  │  火山方舟 API (豆包 2.0 Pro)     │    │
│  └─────────────────────────────────┘    │
│              │                           │
│              ▼                           │
│  ┌─────────────────────────────────┐    │
│  │  飞书开放平台 (消息收发)          │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

**启动方式**：
```bash
# 飞书全流程模式（推荐）
python feishu_ws_client.py

# 传统 GUI 模式
python gui_app.py

# 命令行模式
python feishu_cua.py "帮我预约明天下午3点的会议"
```

**环境要求**：
- Windows 10/11
- Python 3.8+
- 飞书桌面客户端
- `.env` 文件配置飞书应用凭证
- `config_api.json` 配置火山方舟 API


### 4.5 安全机制

| 层级 | 机制 | 说明 |
|------|------|------|
| 物理层 | 鼠标移到左上角 | PyAutoGUI 内置故障安全，紧急停止 |
| 逻辑层 | 最大迭代 250 次 | 防止 AI 陷入无限循环 |
| 行为层 | 连续相同操作检测 | 鼠标操作超 3 次相同 → 强制暂停求助 |
| 网络层 | 指数退避重试 | API 429 限流自动等待重试（最多 5 次） |
| 用户层 | 暂停按钮 | 随时可停，不丢失上下文 |
